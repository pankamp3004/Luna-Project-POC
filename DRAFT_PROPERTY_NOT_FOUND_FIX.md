# DRAFT Mode Enhancement - Property Not Found Detection

## Enhancement Added

**New DRAFT Rule:** When a prospect asks about a property that doesn't exist in our database, the reply is now automatically saved as **DRAFT** instead of being sent.

---

## Why This Matters

### Before This Fix:
If a prospect asked about a non-existent property (e.g., wrong address, property sold, not in our portfolio):
- AI might reply: "Sorry, we don't have that property available"
- This reply would be **SENT** automatically
- Could create confusion or poor customer experience

### After This Fix:
- AI generates a reply about the non-existent property
- System detects tool returned `found: False`
- Reply is automatically saved as **DRAFT**
- Human can review and provide proper response
- Better customer experience and accuracy

---

## How It Works

### Detection Logic

When any tool returns `{"found": False, ...}`, the reply is marked as DRAFT.

**Tools that return `found: False`:**
1. `fetch_property_data` - When property not in master CSV
2. `get_unit_availability` - When property not in YARDI data
3. `check_property_status` - When property not in status CSV

### Code Changes

**File:** `luna_agent.py`

**Function Updated:** `_check_unverified_sensitive_facts()`

**New Parameter Added:** `tool_results: list = None`

**New Rule Added:**
```python
# Rule 2: Check if any tool returned "found: False"
if tool_results:
    for result in tool_results:
        if isinstance(result, dict) and result.get("found") is False:
            _log("[DRAFT] Reply references property not found in our data")
            return True
```

**Tracking Added:**
```python
# In process_email function:
tool_results_data = []  # Track actual tool results

# When tools are called:
tool_results_data.append(result_data)

# When checking for DRAFT:
_check_unverified_sensitive_facts(reply_text, tools_called, tool_results_data)
```

---

## Updated DRAFT Rules

The reply is marked as **DRAFT** when:

| Rule | Condition | Example |
|------|-----------|---------|
| **Rule 1** | Reply mentions rent/pricing WITHOUT calling `get_unit_availability` | "The rent is $1500" but tool not called |
| **Rule 2** | Any tool returns `found: False` (property not in database) | Prospect asks about "123 Fake Street" - property doesn't exist |

## Test Cases

### Test 1: Property Not Found
**Email:**
```
Subject: Inquiry about 999 Nonexistent Ave
Body: Hi, I'm interested in the apartment at 999 Nonexistent Ave. Is it available?
```

**Tool Call:**
```python
fetch_property_data("999 Nonexistent Ave")
# Returns: {"found": False, "message": "No property found matching '999 Nonexistent Ave'"}
```

**Result:**
- ✅ Reply generated by AI
- ✅ System detects `found: False`
- ✅ Decision: **DRAFT**
- ✅ Status: `draft`
- ✅ Reply saved but NOT sent

---

### Test 2: Property Exists (Normal Flow)
**Email:**
```
Subject: Inquiry about 508 W Brighton Ave
Body: Hi, I'm interested in Unit 16 at 508 W Brighton Ave. What's the rent?
```

**Tool Call:**
```python
get_unit_availability("508 W Brighton Ave", unit="16")
# Returns: {"found": True, "property": "508 W Brighton Ave", "requested_unit": {...}}
```

**Result:**
- ✅ Reply generated by AI with actual rent data
- ✅ System detects `found: True`
- ✅ Decision: **SEND** (or HOLD if outside business hours)
- ✅ Reply sent normally

---

### Test 3: Unverified Pricing (Existing Rule)
**Email:**
```
Subject: How much is rent?

**AI Reply:**

Hi! Our typical rent ranges from $1200-$1800 per month.
```

**Result:**
- ✅ Reply mentions pricing
- ❌ But `get_unit_availability` was NOT called
- ✅ Decision: **DRAFT** (Rule 1)
- ✅ Reply saved but NOT sent
